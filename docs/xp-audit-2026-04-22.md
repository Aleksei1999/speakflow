### Пробел #1 — lesson_completed
Когда урок 1-on-1 переходит в `completed`, XP не начисляется. Спека требует 40/55 XP в зависимости от продолжительности. Рекомендуемая реализация — PL/pgSQL trigger на `lessons` (AFTER UPDATE WHEN OLD.status <> 'completed' AND NEW.status = 'completed'), который пишет `xp_events` с amount = `CASE NEW.duration_minutes WHEN 45 THEN 40 WHEN 60 THEN 55 ELSE ROUND(duration_minutes * 55 / 60.0) END`.

### Пробел #2 — клубы
Writer нужен в трёх точках:
1. Регистрация → `club_joined`. Спека не говорит о XP за регистрацию — скорее всего ноль (чтобы не фармили). Рекомендация: **не начислять** за `club_joined`, только за `club_attended`.
2. Атенданс (`club_registrations.status → attended`) → XP по типу клуба (30/40/20/35). Writer-а **нет**. Нужен либо cron (через X часов после `clubs.starts_at` → пометить участников, кто подтвердил присутствие), либо ручной флаг хоста.
3. `club_completed` — опционально, пока не используется.

### Пробел #3 — homework
`/api/lesson/homework` POST создаёт запись с `status='pending'`, но нет PATCH/submit-роута, который бы переводил в `completed` и проверял >80% правильных ответов. Сама фича ещё не достроена.

### Пробел #4 — level_test
`/api/level-test/submit` пишет `level_tests`, но не создаёт `xp_events` c `source_type='level_test'`. Нужно добавить insert при успешной submission.

## 3. Дневные лимиты — ❌ НЕ РЕАЛИЗОВАНЫ

Спека:
- Макс 5 уроков/день за XP
- Макс 2 клуба/день
- 1 daily challenge/день
- 1 тест уровня / 30 дней
- 10 invites max

**В коде ничего этого не проверяется.** Если писатели XP будут добавлены без лимитов, ученик сможет накрутить XP:
- бронируя 10 уроков в день (хотя бизнес-лимит уже мягко есть через слоты),
- проходя тест уровня раз в день (`level-test/submit` не проверяет дату последнего теста),
- приглашая unlimited друзей.

### Рекомендация
Централизовать логику в SQL-функции `award_xp(user_id, source_type, source_id, amount)` с внутренней проверкой дневных лимитов:

```sql
CREATE OR REPLACE FUNCTION award_xp(
    p_user_id UUID, p_source TEXT, p_source_id UUID, p_amount INT
) RETURNS BOOLEAN AS $$
DECLARE
    v_today_count INT;
    v_limit INT;
BEGIN
    v_limit := CASE p_source
        WHEN 'lesson_completed' THEN 5
        WHEN 'club_attended' THEN 2
        WHEN 'daily_challenge' THEN 1
        ELSE 9999
    END;
    SELECT COUNT(*) INTO v_today_count
      FROM xp_events
     WHERE user_id = p_user_id
       AND source_type = p_source
       AND created_at >= date_trunc('day', now());
    IF v_today_count >= v_limit THEN RETURN FALSE; END IF;
    INSERT INTO xp_events(user_id, amount, source_type, source_id)
         VALUES (p_user_id, p_amount, p_source, p_source_id);
    RETURN TRUE;
END $$ LANGUAGE plpgsql SECURITY DEFINER;
```

Писатели из роутов всегда идут через эту функцию — лимиты и audit в одном месте.

## 4. Streak — ❌ НЕТ ЛОГИКИ ВООБЩЕ

Схема есть: `user_progress.current_streak`, `longest_streak`, `last_lesson_date` (migration 006).

**Писателей нет — grep по коду `current_streak =` / `SET streak` / `update.*user_progress.*streak` не находит ничего.**

Это значит:
- `current_streak` всегда `0` для всех пользователей.
- Streak-ачивки (`streak_first_flame`, `streak_week_warrior` и т.д.) никогда не выполнятся.
- Streak-reward-criteria в `rewards` (`{"streak": 14}`) никогда не сработают.

### Спека streak
1. `+5 XP` каждый день активности (любой XP-ивент засчитывается как активность).
2. Milestone: каждые 7 дней подряд — бонус `20 XP`, удваивающийся каждые 30 дней подряд: 20→40→80.
3. Сбрасывается при пропуске дня.

### Рекомендуемая реализация
Cron на `xp_events` (pg_cron, ежечасно или ежесуточно):
```sql
-- Обновить current_streak на основе последних xp_events per user
-- Или: после каждого INSERT в xp_events — триггер, который:
--   если last_xp_date = yesterday → streak += 1, INSERT streak_bonus +5
--   если last_xp_date = today → no-op
--   если last_xp_date < yesterday → streak = 1
```

Проще всего — сделать триггер на `xp_events`, который после успешного insert пересчитывает `current_streak`, `longest_streak`, `last_lesson_date` и, если условия выполнены, делает рекурсивный insert в xp_events с `source_type='streak_bonus'`. Нужна защита от бесконечной рекурсии (trigger level check).
